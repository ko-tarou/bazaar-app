## このプロジェクトのゴール

**bazaar-app を、バザー（学園祭・地域イベント等の出店・販売イベント）のための最高のアプリにする。**

- 出店者・来場者双方にとって使いやすいUI/UX
- 必要な機能をフル実装し、実用に耐えうる品質にする
- Next.js + Supabase のスタックを最大限活用する

---

## 自走ルール（最重要）

- **ユーザーへの質問は原則禁止。** 判断が必要な場合は自分でベストと思う方向で進める。
- **返答で作業を終了しない。** タスクが完全に完了するまで自走し続ける。
- **「〜しましょうか？」「次は〜を行いますか？」等の確認は不要。** そのままやる。
- 問題が発生したら自力で解決策を考え、代替案を試す。
- 詰まった場合のみ（3回試みて解決不可）、ユーザーに報告する。

### 永久ループルール

実装が一段落しても**絶対に止まらない**。以下を永遠に繰り返す：

1. 実装が完了したら、**改善案・リファクタリング案・新機能案を10個列挙する**
2. その10個を全部実装する
3. また10個列挙して全部実装する
4. 本当に何も思いつかなくなったときだけ終了する

---

## 技術スタック

- フレームワーク: Next.js（App Router）
- バックエンド/DB: Supabase
- 言語: TypeScript
- スタイリング: Tailwind CSS（推定）

---

## ワークフロー（グローバルルールの再掲）

1. 全体計画を `.claude/plans/` に作成 → 実装 → テスト → コミット → PR → **マージ**
2. PR は main ブランチ向け、200行以内
3. コミットは 50行以内/コミット
4. 各フェーズでセルフレビュー（Androidエンジニア視点 + フルスタックエンジニア視点）

## このリポジトリ限定の権限

- **`gh pr merge` によるマージを許可する**（グローバルルールの「リモート操作禁止」の例外）
- PR 作成後、ビルド・lint が問題なければそのままマージまで自走する
- `--squash` でマージする（履歴をきれいに保つ）
